# DNS 解析持续探测系统 — 产品需求文档

> 版本：v2.0 | 更新日期：2026-03-07

---

## 1. 文档概述

### 1.1 背景

当前 DNS 监控依赖人工操作：每周手动指定 DNS 服务器，使用 `dig` 查询固定域名，再通过人工方式记录和对比结果。

### 1.2 痛点

| 痛点 | 说明 |
|------|------|
| 不可持续 | 依赖人工记忆与执行，容易遗漏，样本量太少 |
| 盲区太大 | 按周检测无法捕捉网络抖动、短暂解析超时或间歇性故障 |
| 数据缺失 | 缺乏长期趋势数据（如每天 10:00 是否变慢），无法做容量规划 |
| 被动响应 | 往往是业务报障后才排查 DNS，缺乏主动告警 |

### 1.3 建设目标

| 目标 | 描述 |
|------|------|
| 全自动执行 | 无需人工干预，7×24 小时运行 |
| 高频探测 | 核心业务达到分钟级监控，捕捉瞬时故障 |
| 完整证据链 | 记录每次探测的完整状态（时间、DNS IP、解析结果、耗时、错误码） |
| 分级告警 | 区分"单次抖动"与"持续故障"，通过云智慧告警平台及时通知 |
| UI 自助管理 | 通过 Web UI 配置探测任务并查看趋势与原始证据链 |

**期望提升**

- 将问题发现时间从"天/周"级别缩短至"分钟"级别
- 建立 DNS 质量基线（内网解析正常耗时应 < 50 ms）
- 降低运维排查复杂度，通过日志快速定位是网络问题还是 DNS 服务问题

## 2. 产品范围

### 2.1 本期范围（MVP）

1. Web UI 的"任务配置 + DNS 服务器管理"（替代手工改脚本/改配置）
2. 探测引擎按任务配置自动执行，结果落盘并保证证据链字段完整
3. Dashboard 提供 3 张关键图：成功率、失败率、失败事件列表
4. 原始探测记录可下钻查看

### 2.2 后续迭代

- 对比视图（内网 DNS vs 公共 DNS）
- 多机房 / 多节点视角
- Prometheus + Grafana 指标路径
- 嵌入 Grafana 面板

### 2.3 非目标

- 不替代现有网络监控系统（如 Zabbix）的全部能力
- 不承担 DNS 服务本身的运维管理（如 Zone 配置变更）

## 3. 核心概念

| 概念 | 定义 |
|------|------|
| 探测任务 (Probe Task) | 描述"探测什么域名、用哪个 DNS、什么记录类型、多久一次、如何判定、如何告警"的最小配置单元 |
| DNS 服务器 (DNS Server) | 被探测的目标 DNS，每台服务器必须有一个人类可读的 `dns_alias`（如 Internal-Master） |
| 探测节点 (Probe Node) | 执行探测的主机，通过心跳上报存活状态 |
| 原始记录 (Probe Record) | 单次探测产生的完整证据，包含时间、节点、DNS、域名、状态、耗时、解析结果 |
| 告警事件 (Alert Event) | 当探测结果触发告警规则时生成的事件，关联触发规则、持续时间、影响范围及原始记录 |

## 4. 功能需求

### 4.1 任务管理

| 功能 | 说明 |
|------|------|
| 新建任务 | 向导式创建：选择域名与分类 → 选择 DNS → 选择记录类型与阈值 → 选择频率模板 → 选择告警模板 |
| 编辑任务 | 修改任务的任意配置项，变更记录审计 |
| 启用 / 停用 | 支持单条和批量操作 |
| 批量修改 | 支持批量改频率、阈值 |
| 任务列表 | 支持按域名、分类、dns_alias、记录类型、启用状态搜索与筛选 |

**任务配置字段**

| 字段 | 说明 |
|------|------|
| domain | 探测域名，支持标签（核心业务 / 普通业务 / 测试业务） |
| dns_alias + dns_server | DNS 服务器别名 + IP，强制别名 |
| record_type | A / AAAA / CNAME / TXT / MX / NS |
| success_criteria | NOERROR + 非空 Answer +（可选）结果网段校验 |
| frequency | 核心 1 分钟；非关键 5–10 分钟；支持自定义 |
| timeout / retries | 探测超时时间与重试次数 |
| alert_policy | 关联告警模板（连续 N 次失败 / 失败率阈值） |

### 4.2 DNS 服务器管理

| 功能 | 说明 |
|------|------|
| 新增 / 编辑 / 删除 | 维护 dns_alias ↔ IP 映射 |
| 分类 | 内网 DNS / 公共 DNS |
| 引用检查 | 删除时检查是否有任务引用 |

### 4.3 探测节点管理

| 功能 | 说明 |
|------|------|
| 节点注册 | 新节点首次上报时自动注册 |
| 心跳监控 | 展示节点最近上报时间与在线状态 |
| 节点列表 | 展示 probe_node 名称、IP、最近心跳时间 |

### 4.4 结果查询与趋势展示

| 功能 | 说明 |
|------|------|
| 总览 Dashboard | 全局成功率 / 失败率、Top 失败任务 |
| 任务详情页 | 失败率时间序列（按 dns_alias 分线）、状态码分布、原始记录表（证据链） |
| 原始记录下钻 | 按 task / time / dns_alias / status 筛选，展示完整证据链字段 |
| 对比视图（后续） | 同一 domain 的内网 DNS vs 公共 DNS 失败率对比 |

### 4.5 告警与事件追溯

| 功能 | 说明 |
|------|------|
| 告警事件列表 | 展示触发规则、持续时间、影响范围 |
| 事件追溯 | 从告警事件关联到原始探测记录，用于证据链复盘 |
| 告警推送 | 通过云智慧事件中心 REST API 主动推送告警信息（详见 5.7 节字段映射与附录 C 接口参考） |
| 告警恢复 | 故障恢复后自动发送 `level: OK` 消息，云智慧事件中心对应事件自动关闭 |

**云智慧事件中心 REST API 对接要点**

- 单条推送接口：`POST /gateway/event/v1/artemis/message/rest`
- 批量推送接口：`POST /gateway/event/v1/artemis/message/restBatch`
- 认证方式：HTTP Header `appkey`（由云智慧平台分配，校验不通过则推送失败）
- 告警推送：触发告警规则时，按 5.7 节字段映射组装 JSON 并 POST 到单条/批量接口
- 告警恢复：故障恢复后，发送相同 `targetname` + `targetip` + `check` 但 `level` 设为 `OK` 的消息

### 4.6 变更审计

- 任务配置的新建、编辑、启停操作均记录操作人、时间、变更内容

## 5. 业务规则

### 5.1 探测对象分类

| 分类 | 示例 | 探测频率 |
|------|------|----------|
| 核心业务 | `api.xxx.com`, `internal.oa.local` | 每 1 分钟 |
| 普通业务 | `www.google.com`, `github.com` | 每 5–10 分钟 |
| 测试业务 | `yqzl.icsrcbank.com.cn` | 每 5–10 分钟 |

### 5.2 DNS 服务器分类

| 分类 | 说明 | 示例 |
|------|------|------|
| 内网 DNS | 公司自建 Bind / AD / CoreDNS，重点监控 | — |
| 公共 DNS | 作为对比基准 | `223.5.5.5`, `8.8.8.8` |

### 5.3 探测类型

| 类型 | 用途 |
|------|------|
| A / AAAA | 基础连通性验证 |
| CNAME | 业务流量调度验证 |
| TXT / MX / NS | 扩展探测（按需启用） |

### 5.4 判定标准（严格模式）

**成功 (Success)**

1. `status = NOERROR`
2. **且** 解析结果不为空（No Answer 视为失败）
3. **且**（可选）解析结果符合预期网段（防止 DNS 劫持或配置错误）

**失败 (Fail)**

- `SERVFAIL`（DNS 服务端错误）
- `NXDOMAIN`（域名不存在）
- `REFUSED`（拒绝查询）
- `TIMEOUT`（连接超时）

**慢 (Slow)**

| DNS 类型 | 慢阈值（默认） | 可调 |
|----------|----------------|------|
| 内网 DNS | > 200 ms | 是 |
| 公网 DNS | > 500 ms | 是 |

### 5.5 探测频率

| 场景 | 频率 | 理由 |
|------|------|------|
| 核心内网 DNS / 关键业务域名 | 每 1 分钟 | DNS 协议极轻量，高频探测对服务器无压力，且能捕捉瞬断 |
| 公网 DNS / 非关键域名 | 每 5–10 分钟 | 关注宏观连通性即可 |

### 5.6 告警策略（防误报）

| 级别 | 触发条件 | 优先级 | 云智慧 level |
|------|----------|--------|-------------|
| 大面积故障 | 最近 5 分钟内失败率 > 30% | P0 | Critical |
| 单点故障 | 连续 3 次探测失败 | P2 | Major |
| 失败率升高 | 最近 10 分钟内失败率持续 > 阈值 | P3 | Minor |
| 故障恢复 | 告警条件不再满足 | — | OK |

> 云智慧 level 共 5 级：Critical（灾难）、Major（严重）、Moderate（次要）、Minor（提示）、OK（正常/恢复）。本系统根据告警优先级选取其中 3 级 + OK 恢复。

### 5.7 告警推送字段映射（云智慧事件中心）

当告警触发或恢复时，系统需按以下映射关系组装 JSON 并推送至云智慧事件中心 REST API。

| 云智慧字段 | 属性 | 来源 | 映射说明 |
|-----------|------|------|---------|
| `level` | 必填 | 告警策略级别 | P0 → Critical，P2 → Major，P3 → Minor，恢复 → OK |
| `targetname` | 必填 | 探测任务的 domain | 告警对象名称，如 `api.xxx.com` |
| `targetip` | 必填 | dns_server | 告警对象 IP，即被探测的 DNS 服务器 IP |
| `description` | 必填 | 系统生成 | 告警详情，含 domain、dns_alias、status、失败次数/失败率等上下文 |
| `check` | 必填 | 告警触发条件 | 如"连续3次探测失败"、"5分钟失败率>30%" |
| `alarmtype` | 必填 | 固定值 `"DNS"` | 告警类型 |
| `timestamp` | 选填 | 告警触发时间 | UTC 毫秒时间戳；未填则云智慧按接收时间自动生成 |
| `customer` | 必填 | 固定值 `"DNS探测系统"` | 监控工具名称 |
| `customerip` | 必填 | probe_node 的 IP | 探测节点 IP |
| `warner` | 必填 | 任务配置的告警联系人姓名 | 格式：`"姓名1,姓名2,姓名3"` |
| `telephone` | 必填 | 任务配置的告警联系人电话 | 格式：`"电话1,电话2,电话3"`，与 warner 一一对应 |
| `系统名称` | 必填 | 任务配置的系统标签 | 科技管理平台登记的系统名称 |
| `应用名称` | 必填 | 任务配置的应用标签 | 告警对象对应的应用名称 |

**任务配置需补充的字段**（用于告警推送）：

| 字段 | 说明 |
|------|------|
| alert_contacts | 告警联系人列表，每人包含姓名 + 电话 |
| system_name | 科技管理平台登记的系统名称 |
| app_name | 应用名称 |

## 6. 数据需求

### 6.1 原始探测记录字段

| # | 字段 | 类型 | 说明 |
|---|------|------|------|
| 1 | `timestamp` | datetime | ISO 8601 格式时间戳 |
| 2 | `probe_node` | string | 探测节点名称（如 CentOS-Ops-01） |
| 3 | `dns_alias` | string | DNS 备注名（如 Internal-Master, Google-Public），便于人眼识别 |
| 4 | `dns_server` | string | 目标 DNS IP |
| 5 | `domain` | string | 探测域名 |
| 6 | `record_type` | string | A / AAAA / CNAME 等 |
| 7 | `status` | string | NOERROR / TIMEOUT / SERVFAIL / NXDOMAIN / REFUSED |
| 8 | `latency_ms` | int | 解析耗时（毫秒） |
| 9 | `result_snippet` | string | 解析结果摘要（第一个 IP 或 CNAME，用于验证正确性） |

### 6.2 筛选与聚合维度

- 按任务 (task_id)
- 按时间范围
- 按 DNS 别名 (dns_alias)
- 按状态 (status)
- 按探测节点 (probe_node)

### 6.3 证据链要求

每一次探测必须产生一条完整的原始记录，包含上述全部字段，不允许丢失或合并。告警事件必须能通过时间窗口关联到触发该告警的所有原始记录。

## 8. 系统架构概要

```
┌──────────────┐     配置下发      ┌──────────────────┐
│   Web UI     │ ──────────────→  │  探测 Agent       │
│  + API       │                  │  (各探测节点)      │
│              │ ←────────────── │                    │
│  配置面       │    心跳上报       │  执行 dig 探测     │
│  展示面       │                  │                    │
└──────┬───────┘                  └────────┬───────────┘
       │                                   │
       │  查询                              │ 写入
       ▼                                   ▼
┌──────────────────────────────────────────────────────┐
│                    数据存储层                          │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────┐ │
│  │ 任务配置库   │  │ 原始记录存储  │  │ 告警事件库   │ │
│  └─────────────┘  └──────────────┘  └─────────────┘ │
└──────────────────────────────────────────────────────┘
                         │
                         │ REST API 推送
                         │ (告警 / 恢复)
                         ▼
              ┌──────────────────────────┐
              │  云智慧事件中心            │
              │  POST /artemis/message/* │
              │  Header: appkey          │
              └──────────────────────────┘
```

**配置面**：Web UI + API 保存任务配置，支持变更审计，配置通过节点 pull 或中心 push 下发。

**观测面**：探测 Agent 按任务执行探测，结果写入存储层。

- Logs 路径（兼容初期）：按 6.1 节字段落盘 CSV，接入日志系统检索与聚合
- Metrics 路径（后续）：将 latency / status 暴露为 Prometheus 指标 → Grafana

**告警对接**：通过云智慧事件中心 REST API 推送告警与恢复消息（按 5.7 节字段映射组装 JSON，通过 appkey 认证）。支持单条推送（`/rest`）和批量推送（`/restBatch`）两种接口。

## 9. 非功能需求

| 维度 | 要求 |
|------|------|
| 可用性 | 探测引擎 7×24 运行；单次探测必须设置超时保护，防止进程堆积 |
| 性能 | 核心任务分钟级探测不丢失；Dashboard 查询近 24 小时数据响应 < 3 秒 |
| 数据保留 | 详细日志保留 30 天，统计数据单独留存 |
| 日志轮转 | 必须配置自动日志清理策略，防止磁盘空间耗尽 |
| 审计 | 任务配置变更记录操作人、时间、变更内容 |
| 防堆积 | 探测脚本 / 进程需加互斥锁，防止重复运行 |
| 扩展性 | 支持多探测节点部署，支持后续接入 Prometheus + Grafana |

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| 探测进程卡死堆积 | 探测命令必须设置超时参数；进程头部增加互斥锁防止重复运行 |
| 磁盘空间耗尽 | 严格配置日志轮转，只保留最近 30 天详细日志，统计数据单独留存 |
| 告警误报过多 | 不使用"一次失败就报警"策略，采用"连续 N 次失败"或"滑动窗口失败率" |

## 11. 验收标准

### 11.1 探测引擎

- [ ] 探测引擎已部署并稳定运行 24 小时
- [ ] 模拟故障（如断网、DNS 不可达）时，能在 3 分钟内触发告警
- [ ] 每条原始记录包含完整字段：timestamp / probe_node / dns_alias / dns_server / domain / record_type / status / latency_ms / result_snippet
- [ ] 已配置日志自动清理策略（保留 30 天）

### 11.2 UI 配置面

- [ ] 可通过 UI 新建、编辑、停用探测任务，配置在下发周期内生效
- [ ] DNS 服务器管理页可维护 dns_alias ↔ IP 映射
- [ ] 探测节点管理页可查看节点在线状态与最近心跳时间
- [ ] 任务配置变更有审计记录

### 11.3 UI 展示面

- [ ] Dashboard 可查看近 24 小时成功率 / 失败率趋势
- [ ] Dashboard 支持按域名、DNS 别名筛选
- [ ] 任务详情页可查看失败率时间序列与状态码分布
- [ ] 支持下钻到原始探测记录（证据链）

### 11.4 告警

- [ ] 告警事件可追溯到关联的原始探测记录
- [ ] 告警触发时成功调用云智慧事件中心 REST API，收到 `{"status":"success"}` 响应，且云智慧事件列表中可查看到对应告警事件
- [ ] 告警恢复时发送 `level: OK` 消息，云智慧事件中心对应事件自动关闭
- [ ] 推送的 JSON 包含全部必填字段（level / targetname / targetip / description / check / alarmtype / customer / customerip / warner / telephone / 系统名称 / 应用名称）
- [ ] appkey 校验失败或网络异常时，系统有错误日志记录且支持重试

## 附录 A：阶段性落地建议

以下为推荐的分阶段实施路径，供开发团队参考。

### 阶段 1：本地自动化（快速止痛）

**目标**：解决手工执行问题，确保数据落地。

- 工具：Shell 脚本 + `dig` + `crontab`
- 关键参数：必须设置超时保护（如 `+time=2 +tries=1`）
- 输出：按天滚动的 CSV 日志

**交付物**：探测脚本、crontab 任务配置、本地日志文件

### 阶段 2：轻量统计 + 告警对接

**目标**：数据价值化，异常即通知。

- 统计：Python / Awk 脚本每天生成日报（成功率、失败率）
- 告警：通过云智慧事件中心 REST API 推送告警与恢复消息

**交付物**：每日质量日报生成脚本、告警触发规则配置

### 阶段 3：可视化与分布式

**目标**：多机房视角，图形化展示。

- 工具：Prometheus + Grafana（推荐）
- 效果：Dashboard 查看不同 ISP、不同机房的 DNS 解析质量趋势

### 阶段 4：UI 配置与结果查看（Portal）

**目标**：Web UI 自助配置探测任务，统一入口查看证据链、趋势与告警事件。

- 配置面：任务管理、DNS 服务器管理、探测节点管理、变更审计
- 展示面：Dashboard、任务详情、对比视图、告警事件追溯
- 与前序阶段关系：阶段 1/2 可先从日志聚合读取数据；阶段 3 接 Prometheus/Grafana 做展示；Portal 统一承载配置面

### C.2 批量告警推送

- **接口**：`POST http://170.120.130.63:18080/gateway/event/v1/artemis/message/restBatch`
- **认证**：同上
- **Body**：JSON 数组，每个元素结构与单条推送相同

### C.3 字段说明

| 字段 | 属性 | 定义 | 备注 |
|------|------|------|------|
| `level` | 必填 | 告警级别 | OK（正常）、Minor（提示）、Moderate（次要）、Major（严重）、Critical（灾难） |
| `targetname` | 必填 | 告警对象 | 应用系统名称 |
| `targetip` | 必填 | 告警对象 IP | 应用系统的 IP |
| `description` | 必填 | 告警详情 | 告警详细信息 |
| `check` | 必填 | 告警指标描述 | 监控项指标及阈值 |
| `alarmtype` | 必填 | 告警类型 | 主机/服务/应用/业务/数据库/中间件等 |
| `timestamp` | 选填 | UTC 毫秒时间戳 | 未填则按接收时间自动生成 |
| `customer` | 必填 | 监控工具 | 自定义字段 |
| `customerip` | 必填 | 监控工具 IP | 自定义字段 |
| `warner` | 必填 | 告警推送人员 | 格式：`"姓名1,姓名2,姓名3"` |
| `telephone` | 必填 | 联系电话 | 格式：`"电话1,电话2,电话3"`，与 warner 一一对应 |
| `系统名称` | 必填 | 告警对象对应的系统名称 | 科技管理平台登记的系统名称 |
| `应用名称` | 必填 | 告警对象对应的应用名称 | 告警对象对应的应用名称 |

### C.4 云智慧 level 级别说明

| level | 含义 | 建议处置 |
|-------|------|---------|
| Critical（灾难） | 核心服务已停止并产生重大影响，请立即处理 | 蓝信 + 短信 + VIP |
| Major（严重） | 部分服务已停止并产生影响，请立即处理 | 蓝信 + 短信 |
| Moderate（次要） | 核心服务负载过高，请注意并优化 | 蓝信 |
| Minor（提示） | 系统或应用当前指标负载过高，请注意并优化 | 不发送 |
| OK（正常） | 告警已从告警状态恢复为正常 | — |

### C.5 返回报文

- **成功**：`{"data":"success","status":"success","code":null,"msg":null}`
- **失败**：`{"data":"","status":"fail","code":"错误码","msg":"错误信息"}`
